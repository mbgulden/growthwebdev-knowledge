# OKF Sprawl Curation (2026-07-01)

## Why this report exists

GRO-3122: my own 2026-07-01 self-audit found that the OKF at
`/home/ubuntu/work/growthwebdev-knowledge/okf/` has accumulated **274 markdown
files** across 23 subdirectories, with **108 of those being `ned-scan-triage-*`
files** I generated in a 4-day window (2026-06-27 → 2026-06-30).

This is a sprawl problem. The OKF is a knowledge base — every file should add
information that isn't already captured elsewhere. Per the OKF spec, knowledge
is captured at the moment it matters; a SUPPRESS outcome with identical content
to the prior tick is **not** new knowledge and is bloating the hub.

This report:
1. Inventories the current state
2. Diagnoses the sprawl pattern
3. Recommends and applies **minimal** curation
4. Establishes a maintenance contract for going forward

## Sprawl diagnosis

1. **Mechanical-SUPPRESS loop (r59):** The Ned scan-triage rule (per GRO-2021 / r59) was
   designed to fire only when a previously-anchored batch changes content. But the
   scanner feed was identical for 17+ consecutive ticks, and Ned was still writing
   one OKF file per tick. **Each file has the same 10-item table, the same SUPPRESS
   outcome, and the same standing-escalation footer.** Only the r-number and the
   timestamp differ.

2. **No frontmatter on scan-triage files:** Per OKF spec, every doc needs `timestamp:`
   in YAML frontmatter. Of 108 Ned scan-triage files, **0** have proper frontmatter
   — they're plain markdown with `# Ned scan triage YYYY-MM-DD rN` as the only
   date-stamp. This violates OKF's date-stamp invariant (see `okf/index.md` §Maintenance).

3. **Master `audits/index.md` is a wall of duplicates:** r1–r138 with ~95% identical
   text. Searching the OKF for "Ned scan triage" returns 108 hits, but the
   information density is closer to 1 hit (the underlying pattern hasn't changed).

4. **No deprecation / supersedes metadata:** A handful of older operational plans
   (e.g., `agy-dispatch-v2-lane-supervisor-2026-06-26.md`) have been superseded
   by newer ones (`agy-os-level-sandboxing-2026-06-27.md` carries an explicit
   `supersedes:` field) but the older doc is still in the index without
   `status: superseded` frontmatter.

## Curation actions taken (2026-07-01, GRO-3122)

- **This report added** at `okf/audits/_curation-2026-07-01.md` (the file you're reading).
- **Master index updated** to reference this report and document the maintenance contract.
- **`ned-okf-curation` script added** at `~/.hermes/profiles/ned/scripts/ned_okf_curation.sh`
  — runs weekly, reports on:
    * Files lacking frontmatter
    * Files with `last_verified` > 60 days ago
    * Sections with > 50% of files dated within 7 days (sprawl signal)
    * Duplicate-content detection (md5 on first 1KB)
- **No files deleted** — curation is non-destructive. Recommendations only.

## Maintenance contract (going forward)

The Ned weekly self-audit (GRO-3129) now includes a step that runs
`ned-okf-curation.sh` and alerts if:
- Any section has > 5% of files without frontmatter, OR
- Any file is older than 90 days without `last_verified` being updated, OR
- The total file count grows by > 10% week-over-week without a corresponding
  growth in `linear_issue` references (suggests uncurated output)
