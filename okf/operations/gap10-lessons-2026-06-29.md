# Gap 10 Lessons — Subagent Delegation With Commit Checkpoints

**Date:** 2026-06-29
**PR:** #47 (open, awaiting Sonnet peer review)
**Outcome:** SUCCESS after 2 attempts

## What Happened

Two subagent attempts to implement Gap 10. First attempt: 50KB of
uncommitted code lost when subagent hit tool-call budget. Second
attempt: clean implementation with 7 partial commits, 13/13 tests
passing, total 545/545 on full repo.

## The Diff Between Attempts

### Attempt 1 (FAILED — work lost)

Subagent was given ONE BIG PROMPT:
- "Implement Gap 10 fully"
- "13 tests, 7 files, README update"
- "Report back at the end"

Result:
- Subagent created 4 files (manifest, plugin.py, README, tests/__init__.py)
- Subagent's tests collected FAIL — `ModuleNotFoundError: No module named 'prismatic_hello_world'`
- Subagent exhausted tool budget debugging the import issue
- ZERO commits made — all work in working tree
- Branch checkout by background Ned agent overwrote my working tree
- All Gap 10 work LOST

### Attempt 2 (SUCCESS — 545/545 tests passing)

Same scope, but the prompt explicitly said:
- "COMMIT AFTER EVERY 2-3 FILES CREATED"
- "Use commits like `wip: created plugin-manifest.yaml`"
- "If you hit the tool budget mid-implementation, the partial work survives"
- "Plugin directory MUST be named `prismatic_hello_world` (underscores)"
- "The plugin directory MUST contain `__init__.py`"

Result:
- Subagent made 7 partial commits, each scoped to 1-3 files
- Even when subagent hit tool budget, the 6 completed commits were preserved
- I (Fred) resumed, did a soft-reset to deploy-fresh, committed everything as one clean "Gap 10" commit
- 545/545 tests passing, lint clean, pushed, PR opened

## The Lesson

**"Commit early, commit often" applies to subagents MORE than to humans.**

A human can use mental context to track uncommitted work. A subagent
LITERALLY cannot — when its tool budget hits, the conversation ends
and the working tree disappears on the next checkout.

## Prompting Patterns That Worked

1. **Explicit checkpoint rules**: "After creating files X and Y, commit with message Z. Do not skip this step."

2. **Specific commit message format**: `wip: created plugin-manifest.yaml` is clearer than "save progress" — the commit log becomes a step-by-step implementation record.

3. **Acknowledge partial completion is OK**: "If you hit the tool budget mid-implementation, the partial work survives." This freed the subagent from trying to do everything in one go.

4. **Pre-state warnings about pitfalls**: "Plugin directory MUST be named `prismatic_hello_world` (underscores) NOT `prismatic-hello-world` (dashes)". Saved an entire debug cycle.

5. **Specific test counts**: "532 baseline, 545 target (+13)". Subagent knows when it's done; vague goals ("make sure tests pass") waste cycles.

## Prompting Patterns That Failed

1. **"Implement Gap 10 fully"**: too vague. Subagent didn't know what "fully" meant, made architectural choices that conflicted with later requirements.

2. **"Report back at the end"**: assumes the subagent will reach "the end". With a 50-call budget, that's not guaranteed.

3. **No commit checkpoints**: even if subagent finishes the implementation, the next checkout will lose it. (Background agents switching branches is a real risk on this repo — saw it happen with the GRO-537 Ned triage agent.)

4. **Mixing "spec reading" with "implementation" in one prompt**: subagent burned ~10 tool calls reading docs before writing code. For multi-hour tasks, split into "research" subagent + "implement" subagent.

## Refined Prompt Template (for future Gap-sized work)

```text
You are implementing Gap X of the [PROJECT] plan.

Repository: [PATH]
Current branch: [BRANCH] (do not switch — background agents may overwrite your work)
Last deploy-fresh: [SHA]
Baseline test count: [N]
Target after Gap X: [N + M]

CRITICAL RULES (learned from previous attempts):
1. [Specific pitfall 1 — e.g. "Directory MUST be named with underscores, not dashes"]
2. [Specific pitfall 2 — e.g. "MUST contain __init__.py"]
3. **COMMIT AFTER EVERY 2-3 FILES CREATED.** Use `wip: ...` commit messages.
4. [Other specific constraint]

DELIVERABLES (N files):
A. [file 1] — [what it does, what it tests]
B. [file 2]
...

VERIFICATION:
[exact bash commands to run]
EXPECTED OUTPUT: [what success looks like]

WORKFLOW:
1. Create file A, commit `wip: created A`
2. Create file B, commit `wip: created B`
3. ...
N. Run verification — if green, commit `feat: Gap X complete`
N+1. Report back

REPORT BACK:
- Test count delta (was [N], now [N+M])
- Files created/modified
- Commit hashes
- Any issues that need parent attention

DO NOT push to origin. DO NOT open a PR.
```

## Subagent Budget Reality

- 50 tool calls per subagent (configured max)
- ~5-7 minutes wall time
- ~30 file writes OR ~15 file writes + 15 verifications
- A 13-test Gap with 7 files = ~20 file writes + ~10 verifications + ~5 commits = 35 tool calls (fits)
- A 30-test Gap with 15 files = ~40 file writes + ~15 verifications + ~10 commits = 65 tool calls (DOES NOT FIT — split into 2 subagents, one for "create files", one for "write tests + verify")

## When to Delegate vs. Do Yourself

**Delegate when:**
- 3+ files to create
- Clear scope, no design decisions
- Tests are routine (mutation-through assertions)
- 50+ tool calls of mechanical work

**Do yourself when:**
- Design decisions need user input
- Cross-cutting changes to multiple modules
- Context window matters (you have more context than the subagent)
- Need to coordinate with parallel background agents
